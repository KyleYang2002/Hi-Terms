# Hi-Terms 愿景文档与初步需求

**English Name:** Hi-Terms Vision and Initial Requirements  
**产品名称:** Hi-Terms  
**文档类型:** 愿景文档 + 初步需求文档  
**语言:** 中文

## 1. 产品愿景

Hi-Terms 的愿景不是做一个单纯对标 iTerm 的通用终端模拟器，而是成为一个面向 AI CLI 工具的 macOS 终端会话宿主。

在 Hi-Terms 中，像 Claude Code 这样的 AI 命令行工具，不再只是运行在终端里的黑盒进程，而是被提升为可观察、可交互、可协作的会话对象。

Hi-Terms 希望解决的问题是：

> 让人、外部工具以及运行在终端中的 AI CLI，能够围绕同一个会话进行自然协作。

## 2. 产品主张

Hi-Terms 的核心产品主张是：

> 外部工具可以和 Hi-Terms 中运行的 Claude Code 等工具进行自然交互。

这里的“自然交互”不是简单模拟键盘输入，也不是通过脆弱的方式去截取终端文本，而是让系统真正理解 Claude Code 所在的会话。

这意味着外部工具应当能够：

- 向 Claude Code 发送提示词或补充上下文
- 查看 Claude Code 的当前输出和阶段性结果
- 获取 Claude Code 的当前状态，例如空闲、执行中、等待输入中
- 在必要时中断、恢复或继续某个会话
- 围绕同一个 Claude Code 会话协作，而不是争抢一个原始终端输入流

## 3. Hi-Terms 与 iTerm 的本质区别

Hi-Terms 和 iTerm 的区别不是“功能多少”的区别，而是“产品对象”的区别。

### iTerm 主要解决的问题

- 人和 shell 的交互
- 通用终端窗口的管理和体验
- 终端仿真、渲染、分屏、搜索、配置、性能和稳定性

### Hi-Terms 主要解决的问题

- Claude Code 等 AI CLI 的会话托管
- 外部工具与 AI CLI 会话的协作式交互
- 会话状态、上下文、输入输出边界和多工具协同

一句话概括：

> iTerm 管理的是终端窗口；Hi-Terms 管理的是 AI 会话。

因此，Hi-Terms 的重点不是先把终端本体做到最强，而是先把 AI CLI 的承载和交互做成一等能力。

## 4. 对 Hi-Terms 的产品理解

在 Hi-Terms 中，Claude Code 不只是一个运行中的命令行进程，而是一个有状态、有上下文、可被外部工具围绕操作的会话主体。

系统需要理解的内容不只是“终端里输出了什么”，还包括：

- 当前运行的是哪个 Claude Code 会话
- 当前目录、上下文和环境是什么
- 它现在是否空闲
- 它是否正在输出结果
- 它是否正在执行命令
- 它是否正在等待用户输入
- 一次 prompt 对应的结果边界在哪里

因此，Hi-Terms 的本质更接近于：

> 一个为 Claude Code 等 AI CLI 设计的会话操作系统 + 终端可视化外壳。

## 5. 初步产品边界

### In Scope

Hi-Terms 第一阶段聚焦以下能力：

- 在 macOS 上稳定运行 Claude Code 等 AI CLI 工具
- 提供长期存活的 terminal session
- 让外部工具可以连接到指定会话
- 支持向会话发送 prompt
- 支持查看输出结果和当前状态
- 支持高层语义交互与底层终端注入并存
- 支持必要的中断、恢复和继续执行能力

### Out of Scope

Hi-Terms 第一阶段不以这些目标为重点：

- 不追求立即全面超越 iTerm
- 不追求一开始就做到最强的终端定制体验
- 不把重点放在大量主题、分屏细节或重度终端个性化能力上
- 不把产品扩展成完整远程运维平台或大而全自动化平台

## 6. 自然交互的定义

对于 Hi-Terms 而言，“自然交互”至少意味着：

- 外部工具知道自己操作的是一个 Claude Code 会话，而不是一块匿名终端缓冲区
- 系统可以识别一次提示词和一次结果之间的边界
- 系统可以判断 Claude Code 当前状态
- 多个工具围绕同一会话协作时，不会互相打断或写乱输入流
- 在能使用高层语义接口时，不必退回到原始终端文本注入
- 在必要时仍保留原始终端注入能力，用于兼容或兜底

## 7. 建议的系统理解

Hi-Terms 可以先按四层结构理解：

### 1) Terminal Runtime

负责基础终端承载：

- PTY
- shell 和子进程运行
- 基础输入输出
- 终端渲染
- tab 或窗口承载

### 2) Session Host

负责 AI CLI 会话生命周期管理：

- 启动 Claude Code
- 维持会话长期存活
- 记录上下文与状态
- 支持暂停、恢复、结束

### 3) Interaction Layer

负责外部工具与会话的交互接口：

- 高层接口，例如 `send_prompt`、`read_output`、`get_state`、`interrupt`
- 底层终端注入能力，作为兼容模式

建议高层接口作为主路径，原始终端注入作为兜底路径。

### 4) Tool Bridge

负责其他 Tools 对 Hi-Terms 的接入：

- 连接会话
- 发送请求
- 获取结果
- 监听状态变化
- 管理并发访问和协作边界

## 8. 初步需求

### 核心需求

- Hi-Terms 必须能够稳定承载 Claude Code 等 AI CLI 工具的运行
- Hi-Terms 必须将 Claude Code 会话视为一等对象，而不只是普通终端进程
- Hi-Terms 必须允许外部工具对会话进行自然交互
- Hi-Terms 必须提供会话状态可见性
- Hi-Terms 必须支持输出读取与结果边界识别
- Hi-Terms 必须支持多个工具围绕同一会话进行协作
- Hi-Terms 必须同时支持高层会话接口和底层终端注入能力

### 体验需求

- 用户应当能明确看到某个 Claude Code 会话当前状态
- 用户应当能知道一次交互对应的结果范围
- 外部工具接入后不应破坏用户在终端中的正常使用体验
- 会话在长时间运行后仍应保持稳定和可恢复

### 系统需求

- 会话必须具备长期存活能力
- 会话状态必须可查询
- 输出流必须可读取和组织
- 并发访问必须有清晰边界
- 系统需要能够处理中断、恢复和继续执行

## 9. 关键挑战

当前方向下，主要难点会集中在：

- 如何准确识别 Claude Code 的会话状态
- 如何划分 prompt 与输出结果的边界
- 如何支持多工具围绕同一会话协作
- 如何避免多个工具同时写入时的冲突
- 如何在高层语义接口和底层兼容能力之间保持平衡
- 当 Claude Code 的交互形式变化时，系统如何保持稳健

## 10. 初步判断

Hi-Terms 不是一个“更强的 iTerm”，而是一个“为 AI CLI 设计的终端会话宿主”。

它的核心竞争力不在于终端仿真本身，而在于：

- 是否能把 Claude Code 会话托管好
- 是否能让外部工具自然接入
- 是否能建立稳定的会话状态、输入输出边界和协作模型

如果这些能力成立，Hi-Terms 就会形成一条与 iTerm 明显不同的产品路线。

## 11. 一句话总结

> Hi-Terms 是一个面向 Claude Code 等 AI CLI 工具的 macOS 会话宿主终端，核心价值在于让外部工具能够与终端中的 AI 会话进行自然协作。
