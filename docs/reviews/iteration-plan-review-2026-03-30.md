# Hi-Terms 迭代计划评审与修订方案

**日期：** 2026-03-30  
**评审范围：** 项目下全部现有文档  
**执行流程：** Clarify → Explore → Plan → Implement → Verify

## 1. Clarify

### 1.1 任务理解

本次任务不是直接改写原有 roadmap，而是基于当前文档体系，对现有迭代计划做一次面向落地实施的评审，重点回答以下问题：

- 当前版本切分是否合理
- 现有里程碑是否支持编码实现落地
- 验收标准是否可验证
- 文档之间是否存在阶段边界或依赖顺序上的冲突
- 如果不调整，后续实现最可能在哪些位置失控

### 1.2 评审标准

本次评审按两个核心维度展开：

- **计划合理性**：版本边界是否清楚，工作量是否匹配，依赖顺序是否正确
- **编码可落地性**：是否具备可实现的基础抽象、验证手段和工程支撑

### 2.2 基线观察

- 产品方向整体一致：第一大阶段先把终端产品做好，第二大阶段再把会话做成一等对象。
- Roadmap 主线没有偏离愿景和需求，但版本切分偏“产品叙事”，偏少“工程落地顺序”。
- 当前仓库仍处于文档阶段，尚无源码、构建系统、测试体系或性能基线。
- 因此，roadmap 中大量“稳定性”“性能”“长期存活”“可扩展”类目标，当前都缺少可验证基础设施。

## 3. Plan

### 3.1 评审结论摘要

当前迭代计划的**方向正确**，但以工程落地视角看，存在四类核心风险：

1. 基础工程能力没有前置，导致后续版本的“稳定性/性能/可回归”目标无从验证。
2. Session Host 的基础抽象放得过晚，和 v0.3 的 AI CLI 稳定承载目标存在依赖倒置。
3. v0.2 负载过重，把终端产品中最难的兼容性工作压缩在同一版本，排期失真风险高。
4. 多个验收标准不可度量或高度主观，后续容易出现“看起来完成了，但无法证明”的情况。

### 3.2 优先级定义

- **P0**：不修订会直接破坏后续版本落地顺序
- **P1**：不修订会明显拖慢节奏或引发大规模返工
- **P2**：不修订不会立刻阻塞，但会持续制造模糊和沟通成本

## 4. Implement

### 4.1 优先级问题清单

#### P0-1 缺少工程基线版本，当前 roadmap 直接从功能交付起步

**证据**

- Roadmap 从 `v0.1` 直接进入 PTY、Shell、渲染能力交付。
- `CLAUDE.md` 明确说明当前仓库仍是 pre-implementation，尚无 build system、tests、source code。
- 第一大阶段和 v0.3/v0.6 又都要求稳定性、性能、长期存活、回归能力。

**问题本质**

这不是普通产品文档问题，而是典型的工程计划缺项。终端产品的核心难点不是“先写出一个窗口”，而是：

- 终端仿真策略怎么选
- 渲染模型怎么定
- Screen buffer/scrollback 数据结构怎么设计
- 自动化回归如何做
- 性能与内存指标怎么采集
- crash/logging 如何支持问题定位

这些能力不先落地，后续每个版本都只能“边做功能边补地基”。

**影响**

- v0.1 会被隐性拆成“做功能 + 搭工程基础设施”两个版本的工作量。
- v0.2 之后的所有验收标准都缺少统一验证机制。
- 性能和稳定性问题会在 v0.3 才集中暴露，修复成本最高。

**修订建议**

在 roadmap 前增加一个 `v0.0 工程基线与技术验证`，至少覆盖：

- macOS 应用骨架与构建链路
- Terminal Runtime 技术选型和最小 spike
- Parser / screen buffer / renderer 的模块边界
- 自动化测试骨架
- 终端兼容性样例集
- 性能采样基线
- 日志与 crash 收集基础

**建议出口标准**

- 可启动最小空应用并跑通开发链路
- 至少有 1 组自动化终端输出样例回放
- 至少有 1 组性能采样结果
- 已确定 Terminal Runtime 和渲染实现路线

#### P0-2 Session Host 基础抽象放在 v0.6，和 v0.3 的目标顺序冲突

**证据**

- v0.3 已要求长会话稳定性、多轮交互、进程树管理、信号传递、等待输入继续。
- 需求文档要求第一大阶段就具备长期存活能力，并在 Session Host 层预留可扩展接口。
- 术语表把 Session Host 定义为跨阶段能力，第一大阶段就应建立基础框架。
- 但 roadmap 把明确的 Session 模型、内部 API、状态维护集中放在 `v0.6`。

**问题本质**

AI CLI 稳定承载不是纯 Terminal Runtime 问题。只要进入“长会话、子进程树、等待输入、状态可见、异常退出处理”，就已经进入 Session Host 基础职责。

如果等到 v0.6 才正式抽象 Session：

- v0.3 很可能先用 tab/pty 直接承载 AI CLI
- v0.4/v0.5 又继续在这种模型上叠功能
- v0.6 再补 Session 抽象时，重构面会横跨 tab、pane、window、状态管理、持久化和日志

**影响**

- 版本依赖顺序被反转
- AI CLI 稳定性验证会建立在不稳固的数据模型上
- Phase 2 的扩展成本显著上升

**修订建议**

把 `Session Foundation` 前移到 `v0.1` 或最晚 `v0.2`，至少先建立：

- `Session ID`
- Session 与 PTY / tab / pane 的映射关系
- 基础生命周期字段
- 进程树跟踪入口
- 基础状态枚举的第一阶段子集
- 内部 registry / query 能力

`v0.6` 不再承担“首次建立 Session 模型”的职责，而是承担“Phase 1 收官与稳定性封版”职责。

#### P1-1 v0.2 范围过大，排期风险明显偏高

**证据**

- v0.2 同时要求 Tab、多窗口、完整 xterm-256color、剪贴板、窗口 resize、shell 退出处理、字体设置、Unicode/CJK/Emoji 宽度。
- 验收里还要求 `git log --graph`、中文宽度正确、一天日常开发可用。

**问题本质**

这些项里至少有三类工作本来就很重：

- **仿真兼容性**：ANSI 属性、256 色、控制序列、resize 行为
- **文本布局正确性**：Unicode/CJK/Emoji 宽度、组合字符、光标定位
- **承载结构**：tab、多窗口、shell 生命周期管理

把它们挤在一个版本，表面上是“从能跑到能用”，实际上会让 v0.2 成为一个超大版本。

**影响**

- v0.2 容易失控并吞掉 v0.3 时间窗口
- AI CLI 差异化验证会被基础兼容性工作延后
- 团队会被迫在兼容性正确性和交付节奏之间做临时妥协

**修订建议**

把 v0.2 拆成两个层次：

- `v0.2 Core Usability`：tab、resize、剪贴板、shell 退出处理、基础字体设置
- `v0.3 Compatibility + AI CLI Foundation`：Unicode/CJK/Emoji 宽度、复杂输出稳定性、AI CLI 长会话与输出性能

如果版本号数量必须保持不变，建议至少把“完整 xterm-256color + Unicode/CJK/Emoji 宽度正确”从原 v0.2 下沉到 AI CLI 稳定版本中统一处理，因为这两类工作与 AI CLI 大量输出、复杂终端内容渲染本身是同一风险域。

#### P1-2 验收标准可验证性不足，存在大量主观项和不可操作项

**证据**

- `不内存泄漏`
- `帧率 ≥ 30fps`
- `内存使用不超过初始值 3 倍`
- `可作为默认终端完成一天常规开发工作`
- `一周无功能缺失导致必须切回 iTerm/Terminal`
- `连续 7 天日常使用无 crash`

**问题本质**

这些标准不是不能用，而是目前没有配套说明：

- 测试环境是什么
- 样本 workload 是什么
- 自动化还是手工
- 指标如何采集
- 结果是否可复现

没有这些上下文，验收结论会变成主观判断。

**影响**

- 版本结束时难以形成统一“是否通过”判断
- 不同成员对完成度理解会分裂
- 未来无法稳定回归

**修订建议**

为每个版本增加统一验收模板：

- 自动化用例
- 手工场景
- 性能指标
- 观测手段
- 退出条件

示例：

- 把“无明显卡顿”改为“在固定 500 行输出样例下，主线程卡顿 > 100ms 的事件数不超过 N 次”
- 把“完成一天常规开发工作”改为“完成规定任务清单：git、vim、top、node/npm、python/pip、Claude Code 基础会话”
- 把“无 crash”改为“基于 dogfooding 日志与 crash 统计，在连续 X 天、Y 小时使用量下无 P0/P1 crash”

#### P2-1 Phase 1 与 Phase 2 的语义边界在 v0.3 有轻度漂移

**证据**

- v0.3 提到“多轮交互支持”“等待输入”“用户澄清后继续”。
- 需求和术语表又明确把高层会话语义、外部应用围绕会话协作定义为第二大阶段核心。

**问题本质**

v0.3 关注 AI CLI 是对的，但表述方式容易让实现提前滑向“识别会话语义”而非“保障终端宿主稳定性”。

**影响**

- 容易提前引入 Phase 2 才需要的复杂度
- 团队可能误把“懂 AI CLI 语义”当成 v0.3 验收重点

**修订建议**

把 v0.3 目标改写为：

- 保证 AI CLI 在终端中稳定运行
- 正确承载等待用户输入与继续执行这一终端行为
- 保障输出、信号、进程、环境兼容性

避免在 Phase 1 文案中提前承诺“会话语义识别”。

#### P2-2 可观测性与回归体系放得过晚

**证据**

- roadmap 到 v0.6 才明确提出日志体系完善、全量回归测试、崩溃上报。
- 但 v0.1 起就已经有 `top`、`vim`、滚动、颜色、长命令、多轮 AI CLI 等复杂场景。

**问题本质**

终端类产品的问题通常是“偶发、依赖输出内容、依赖时序”。没有早期日志和回放机制，后面会很难复现。

**修订建议**

把以下能力前移到 v0.0-v0.2：

- 会话日志与事件日志
- 终端输出样例回放
- 基础回归脚本
- 性能采样面板或 benchmark 命令

### 4.2 修订后的建议版本结构

下面是一版更适合工程落地的 Phase 1 结构。它保留原有产品方向，但调整依赖顺序。

| 建议版本 | 目标 | 关键交付 |
|---|---|---|
| v0.0 | 工程基线与技术验证 | 应用骨架、构建链路、技术选型、测试骨架、日志与性能采样 |
| v0.1 | 终端内核最小闭环 | PTY、shell、parser、screen buffer、基础渲染、输入、滚动、Session 基础映射 |
| v0.2 | 日常可用基础 | tab、resize、剪贴板、shell 生命周期、字体、基础兼容性 |
| v0.3 | 兼容性强化与 AI CLI 稳定承载 | Unicode/CJK/Emoji、复杂输出稳定性、长会话、进程树、信号、环境兼容、Session Host 基础 |
| v0.4 | 高效操作体验 | split pane、搜索、快捷键体系、选择与滚动增强 |
| v0.5 | 个性化与配置 | 主题、Profile、偏好设置、URL、通知、启动配置 |
| v0.6 | Phase 1 收官 | 回归、dogfooding、性能封版、稳定性封版、Phase 2 扩展评审 |

### 4.3 建议前置的核心技术抽象

为了保证后续编码可落地，建议在 roadmap 中显式前置以下抽象，而不是隐含在版本描述里：

#### A. Terminal Core

- ANSI / VT parser
- screen buffer
- attribute model
- scrollback model
- renderer adapter

#### B. Session Foundation

- session identity
- session registry
- session 与 tab/pane/window 映射
- process tree tracking
- state skeleton

#### C. Quality Harness

- fixture playback
- regression scripts
- performance benchmark
- crash/logging pipeline

### 4.4 建议新增的统一出口模板

建议所有版本统一使用以下结构定义验收：

1. 功能出口
2. 自动化出口
3. 性能出口
4. 稳定性出口
5. 手工体验出口

这样 roadmap 才能从“版本愿景描述”升级为“可执行计划”。

## 5. Verify

### 5.1 覆盖性核对

本评审已覆盖：

- 愿景与阶段边界
- 第一大阶段 roadmap 版本切分
- 第一大阶段与第二大阶段依赖关系
- Session Host 与 Terminal Runtime 的架构连续性
- AI CLI 场景与 Phase 边界
- 当前仓库工程现状对 roadmap 的现实约束

### 5.2 最终结论

当前迭代计划**可作为产品方向文档继续存在**，但**不适合作为直接执行的工程排期文档**。如果不修订，最可能出现的问题不是“方向错”，而是：

- v0.2 失控
- v0.3 返工
- v0.6 集中爆雷

最关键的修订动作只有两条：

1. 在 roadmap 前补一个工程基线版本。
2. 把 Session Foundation 和可观测性从 v0.6 前移到前几个版本。

只要这两条调整到位，当前 roadmap 的主体方向仍然成立。
