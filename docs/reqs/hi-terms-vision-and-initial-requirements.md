# Hi-Terms 愿景文档与初步需求

**English Name:** Hi-Terms Vision and Initial Requirements  
**产品名称:** Hi-Terms  
**文档类型:** 愿景文档 + 初步需求文档  
**语言:** 中文

## 1. 产品愿景

Hi-Terms 的愿景不是做一个单纯对标 iTerm 的通用终端模拟器，而是成为一个面向命令行工具会话的 macOS 会话宿主终端。

在 Hi-Terms 中，运行在终端中的命令行工具，不再只是黑盒进程，而是被提升为可观察、可控制、可协作的会话对象。

对于像 Claude Code、Codex CLI 这样的 AI CLI，这种会话化承载会进一步放大价值，因为这类工具本身就具备更丰富的交互语义和长期协作需求。

Hi-Terms 希望解决的问题是：

> 让人、外部工具以及运行在终端中的命令行工具，能够围绕同一个会话进行稳定协作。

## 2. 产品主张

Hi-Terms 的核心产品主张是：

> 外部工具可以围绕 Hi-Terms 中运行的命令行工具会话进行协作式交互。

这里的“协作式交互”不是简单模拟键盘输入，也不是通过脆弱的方式去截取终端文本，而是让系统真正理解某个工具会话的身份、生命周期和输入输出边界。

这意味着外部工具应当能够：

- 连接到指定命令行工具会话
- 查看会话的当前输出和阶段性结果
- 获取会话的当前状态，例如运行中、空闲、已中断、已结束
- 在必要时向会话写入输入，或对支持该模式的工具发送 prompt、补充上下文
- 在必要时中断、恢复或继续某个会话
- 围绕同一个会话协作，而不是争抢一个原始终端输入流

## 3. Hi-Terms 与 iTerm 的本质区别

Hi-Terms 和 iTerm 的区别不是“功能多少”的区别，而是“产品对象”的区别。

### iTerm 主要解决的问题

- 人和 shell 的交互
- 通用终端窗口的管理和体验
- 终端仿真、渲染、分屏、搜索、配置、性能和稳定性

### Hi-Terms 主要解决的问题

- 命令行工具会话的承载与托管
- 外部工具围绕会话的协作式接入
- 会话状态、上下文、输入输出边界和多工具协同

一句话概括：

> iTerm 管理的是终端窗口；Hi-Terms 管理的是命令行工具会话。

因此，Hi-Terms 的重点不是先把终端本体做到最强，而是先把会话承载、会话控制和外部协作做成一等能力。

其中，像 Claude Code、Codex CLI 这样的 AI CLI，会是最核心、最能体现产品差异化价值的使用场景。

## 4. 对 Hi-Terms 的产品理解

在 Hi-Terms 中，命令行工具不只是一个运行中的终端进程，而是一个有生命周期、有上下文、可被外部工具围绕操作的会话主体。

系统需要理解的内容不只是“终端里输出了什么”，还包括：

- 当前运行的是哪个工具会话
- 当前目录、上下文和环境是什么
- 会话是否仍然存活
- 会话是否正在输出结果
- 会话是否正在执行任务
- 会话是否处于可继续、可中断、可恢复的状态
- 对具备相应交互语义的工具，一次 prompt 或一次交互对应的结果边界在哪里

并不是所有 CLI 都具备自然语言交互能力。自然语言能力来自工具本身，而不是终端本体。

因此，Hi-Terms 的本质更接近于：

> 一个面向命令行工具会话的宿主层 + 终端可视化外壳。

## 5. 初步产品边界

### In Scope

Hi-Terms 第一阶段聚焦以下能力：

- 在 macOS 上稳定运行各类命令行工具
- 提供长期存活的 terminal session
- 让外部工具可以连接到指定会话
- 支持查看输出结果和写入输入
- 支持查询会话的基础状态和生命周期
- 支持必要的中断、恢复和继续执行能力
- 支持高层交互与底层终端注入并存
- 优先优化 Claude Code、Codex CLI 等 AI CLI 的交互体验

### Out of Scope

Hi-Terms 第一阶段不以这些目标为重点：

- 不追求立即全面超越 iTerm
- 不追求一开始就做到最强的终端定制体验
- 不把重点放在大量主题、分屏细节或重度终端个性化能力上
- 不把产品扩展成完整远程运维平台或大而全自动化平台

## 6. 高层交互的定义

对于 Hi-Terms 而言，“高层交互”意味着：

- 外部工具知道自己操作的是一个具体的命令行工具会话，而不是一块匿名终端缓冲区
- 系统可以提供稳定的会话连接、输出读取、输入写入和生命周期控制能力
- 多个工具围绕同一会话协作时，不会互相打断或写乱输入流
- 对具备相应语义的工具，系统可以进一步提供状态识别、阶段结果读取和交互边界理解
- 在能使用高层接口时，不必退回到原始终端文本注入
- 在必要时仍保留原始终端注入能力，用于兼容或兜底

这里需要明确：

- Hi-Terms 不负责提供自然语言能力
- 自然语言交互来自 Claude Code、Codex CLI 等工具本身
- Hi-Terms 负责把这类工具的会话托管好，并在可能时暴露更高层的会话接口

## 7. 建议的系统理解

Hi-Terms 可以先按四层结构理解：

### 1) Terminal Runtime

负责基础终端承载：

- PTY
- shell 和子进程运行
- 基础输入输出
- 终端渲染
- tab 或窗口承载

### 2) Session Host

负责命令行工具会话生命周期管理：

- 启动 shell 或目标工具
- 维持会话长期存活
- 记录上下文与基础状态
- 支持暂停、恢复、结束

### 3) Interaction Layer

负责外部工具与会话的交互接口：

- 基础接口，例如 `write_input`、`read_output`、`get_session_state`、`interrupt`
- 对支持该模式的工具提供更高层接口，例如 `send_prompt`
- 底层终端注入能力，作为兼容模式

建议高层接口在适用时作为主路径，原始终端注入作为兜底路径。

### 4) Tool Bridge

负责其他 Tools 对 Hi-Terms 的接入：

- 连接会话
- 发送请求
- 获取结果
- 监听状态变化
- 管理并发访问和协作边界

## 8. 初步需求

### 核心需求

- Hi-Terms 必须能够稳定承载命令行工具的运行
- Hi-Terms 必须将命令行工具会话视为一等对象，而不只是普通终端进程
- Hi-Terms 必须允许外部工具对会话进行连接、读取、写入和控制
- Hi-Terms 必须提供会话生命周期和基础状态可见性
- Hi-Terms 必须支持输出流的读取与组织
- Hi-Terms 必须支持多个工具围绕同一会话进行协作
- Hi-Terms 必须同时支持高层会话接口和底层终端注入能力
- 对具备相应交互语义的工具，Hi-Terms 应支持更高层的状态识别和交互能力

### 体验需求

- 用户应当能明确看到当前是哪个工具会话
- 用户应当能看到某个会话当前的基础状态
- 对支持该模式的工具，用户应当能知道一次交互对应的结果范围
- 外部工具接入后不应破坏用户在终端中的正常使用体验
- 会话在长时间运行后仍应保持稳定和可恢复

### 系统需求

- 会话必须具备长期存活能力
- 会话状态必须可查询
- 输出流必须可读取和组织
- 并发访问必须有清晰边界
- 系统需要能够处理中断、恢复和继续执行
- 系统需要能够在通用会话能力之上扩展工具相关的增强能力

## 9. 关键挑战

当前方向下，主要难点会集中在：

- 如何为不同类型的命令行工具提供统一的会话宿主模型
- 如何准确识别特定工具，尤其是 AI CLI 的会话状态
- 如何划分具备对话语义的工具的一次交互与结果边界
- 如何支持多工具围绕同一会话协作
- 如何避免多个工具同时写入时的冲突
- 如何在通用会话接口和工具相关高层接口之间保持平衡
- 当 Claude Code、Codex CLI 等工具的交互形式变化时，系统如何保持稳健

## 10. 初步判断

Hi-Terms 不是一个“更强的 iTerm”，而是一个“面向命令行工具会话的终端会话宿主”。

它的核心竞争力不在于终端仿真本身，而在于：

- 是否能把命令行工具会话托管好
- 是否能让外部工具稳定接入同一个会话
- 是否能建立清晰的会话状态、输入输出边界和协作模型
- 是否能对 Claude Code、Codex CLI 等高价值工具提供更高层的会话能力

如果这些能力成立，Hi-Terms 就会形成一条与 iTerm 明显不同的产品路线。

## 11. 一句话总结

> Hi-Terms 是一个面向命令行工具会话的 macOS 会话宿主终端，核心价值在于让外部工具能够围绕终端中的会话进行协作，其中 Claude Code、Codex CLI 等 AI CLI 是最重要的高价值场景。
